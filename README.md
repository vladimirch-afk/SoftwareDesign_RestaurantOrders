# Чечуров Владимир БПИ227, КПО ДЗ-2 "Система управления заказами в ресторане"

## О работе

Данное приложение реализует консольное взаимодействие с пользователем, был реализован функционал на 10 баллов, программа написана с применением принципов ООП и SOLID, для приготовления блюд использовалось многопоточное программирование. Использовались шаблоны проектирования. Для хранения данных и удобства чтения данные об аккаунтах, счетах, блюдах и заказах хранятся соответственно в `accounts.json`, `bills.json`, `dishes.json`, `orders.json` находящихся в папке `res` проекта.

## Туториал

При запуске будет выведено следующее меню:

```
Select the option:
1 - Sign in
2 - Create account
3 - Exit program
```

При неудачной загрузки информации из файлов предупреждение появится здесь же, неудачным оно считается при невозможности доступа к файлам, или если файлы пустые.

Процесс регистрации выглядит следующим образом:

```
Enter login:
5
Enter password:
5
```

#### Завершение работы программы

Для завершения работы программы необходимо в меню авторизации выбрать опцию `3 - Exit program`, тогда программа завершится, перед этим сохранив все произведенные изменения в аккаунтах, счетах, меню, заказах. Без такого завершения все изменения не сохранятся.

## Паттерны и приготовление заказов

#### Приготовление:

После создания заказа он отправляется на кухню (экземпляр класса `KitchenServiceImpl()`), которая их ставит в очередь на приготовление, когда освобождается один из 10 доступных поваров (экземпляр класса `CookerImpl()`), то ему передается этот заказ на приготовление (здесь работает система приоритетов: следующим готовится заказ с самой большой выручкой), в результате чего изменяется статус заказа на COOKING (когда заказ только создан он CREATED), повар запускает поток на приготовление заказа, все блюда в заказе он приготавливает по очереди. После завершения приготовления, повар оповещает кухню о готовности заказа.

#### Паттерны:

В реализации программы использовались следующие паттерны:

1. *Шаблонный метод*. Большинство классов в программе наследуются от соответствующих интерфейсов для предоставления возможности в дальнейшем реализовать другие классы, реализующие нужный функционал без надобности переделывать остальной код.
2. *Фабрика*. Используется для создания разных типов аккаунтов (аккаунт администратора или посетителя).
3. *Фасад*. Данный паттерн использовался для сокрытия процесса приготовления заказа, которым управлял `KitchenServiceImpl` (создание заказа, помещение в очередь ожидания, расставление приоритетов, распределение заказов между поварами, оповещение других классов о готовности заказа для его сохранения и выставления счета), благодаря этому для приготовления заказа нужно было вызвать только один метод у экземпляра этого класса. 

## Дополнительные особенности поведения программы и ее реализации

1. У одного посетителя может быть только один заказ, который передали на кухню, если у него имеется не завершенный заказ, который не готов, ему не дадут сделать еще один, необходимо дождаться завершения.
2. После готовности заказа необходимо оплатить счет, если он не оплачен пользователю не дадут сделать еще один заказ.
3. Все данные загружаются из файлов и сохраняются в них только на этапах запуск/завершение программы.
4. По умолчанию на кухне 10 поваров, их количество задается переменной `maxCookers` в классе `KitchenServiceImpl()`.
5. Логин и пароль не могут быть пустыми или состоять только из пробелов, при создании нужно иметь в виду, что все пробелы из них стираются (логины `user    _1` и `user_1` одинаковы).
